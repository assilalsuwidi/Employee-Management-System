# 2. قاعدة البيانات والعلاقات والأمان (Database ERD & Security)

في هذا القسم سنشرح بنية قاعدة البيانات (MySQL) والجداول الأساسية (Tables) وعلاقتها مع بعضها البعض (ERD - Entity Relationship Diagram)، بالإضافة إلى تقنيات الأمان والحماية التي تم تطبيقها لمنع الاختراقات.

---

## 🗄️ جداول قاعدة البيانات والعلاقات (ERD & Models)

تم تصميم قاعدة البيانات لتكون متعادلة (Normalized) لتقليل التكرار وتسهيل التحديثات. الجداول الأساسية تشمل:

### 1. جدول `users` (المستخدمين)
- **الهدف**: تخزين بيانات تسجيل الدخول للمستخدمين (سواء كانوا مدراء أو موظفين).
- **الحقول الأساسية**: `id` (مفتاح أساسي)، `username`، `password_hash`، `role` (صلاحية المستخدم: admin، hr، employee).
- **العلاقات**: يرتبط بجدول `employees` بعلاقة (One-to-One)، حيث أن كل مستخدم في النظام يمثل موظفاً واحداً.

### 2. جدول `employees` (الموظفين)
- **الهدف**: تخزين البيانات الشخصية والوظيفية لكل موظف.
- **الحقول الأساسية**: `id`، `first_name`، `last_name`، `email`، `phone`، `join_date`، `status`.
- **العلاقات**:
  - (Many-to-One) مع جدول `departments`: كل موظف ينتمي لقسم واحد.
  - (One-to-Many) مع جدول `attendance`: الموظف لديه عدة سجلات حضور.
  - (One-to-Many) مع جدول `payroll`: الموظف لديه عدة سجلات رواتب شهرية.

### 3. جدول `departments` (الأقسام)
- **الهدف**: تنظيم الموظفين في هياكل إدارية (مثل: قسم تقنية المعلومات، قسم الموارد البشرية).
- **العلاقات**: (One-to-Many) مع `employees`.

### 4. جداول الحضور والانصراف (`attendance`, `holidays`, `weekends`)
- **`attendance`**: يسجل متى قام الموظف بتسجيل الدخول (`check_in`). ويرتبط بالموظف `employee_id`. تم وضع قيد (Unique Constraint) على (employee_id, date) لمنع الموظف من تسجيل حضوره مرتين في نفس اليوم.
- **`holidays` و `weekends`**: جداول تحتوي على أيام العطل الرسمية وأيام عطلة نهاية الأسبوع التي يستند إليها النظام لمعرفة ما إذا كان اليوم يوم عمل فعلي.

### 5. جداول المهام والمجموعات (`tasks`, `groups`, `group_members`)
- **`groups`**: المجموعات أو الفرق التي يتم تشكيلها لتنفيذ مشاريع محددة.
- **`group_members`**: جدول وسيط (Junction Table) لتكوين علاقة (Many-to-Many) بين الموظفين والمجموعات.
- **`tasks`**: مهام موكلة للموظفين مرتبطة بـ `employee_id`.

### 6. جدول سجل التدقيق (`audit_logs`)
- **الهدف**: جدول مخصص لغايات الأمان (Security Audit) يقوم بتسجيل كل حركة تحدث في النظام مع تاريخ ووقت حدوثها، رقم المستخدم، ونوع العملية (مثال: "قام المدير X بتغيير راتب الموظف Y").

## 🔒 أمان قاعدة البيانات وحمايتها من الثغرات

لقد تم حماية قاعدة البيانات عبر طبقات متعددة من الأمان ضد أكثر الثغرات شيوعاً (بحسب تصنيف OWASP):

### 1. الحماية من ثغرات الحقن (SQL Injection - SQLi)
- **المشكلة**: المهاجم يحاول كتابة أكواد SQL خبيثة داخل حقول الإدخال لتدمير أو سرقة قاعدة البيانات.
- **الحل المُطبق**: لم نستخدم استعلامات SQL نصية مباشرة إطلاقاً. بدلاً من ذلك، نستخدم **SQLAlchemy ORM**. يقوم هذا الـ ORM بتمرير المدخلات كـ (Parameterized Queries) مما يجرد أي كود خبيث من فعاليته ويحوله إلى نص عادي.

### 2. حماية بيانات المرور (Password Cryptography)
- **المشكلة**: تخزين كلمات المرور كنص صريح يعرض كل المستخدمين للخطر في حال تسرب قاعدة البيانات.
- **الحل المُطبق**: نستخدم مكتبة `Bcrypt`. تقوم هذه الخوارزمية بتحويل كلمة المرور إلى سلسلة معقدة (Hash) مع إضافة قيمة عشوائية (Salt). حتى لو كان لدى موظفين اثنين نفس كلمة المرور، سيكون الـ Hash مختلفاً، ومن المستحيل عكسياً فك تشفيره.

### 3. الحماية من التصعيد المفرط للصلاحيات (Privilege Escalation)
- **المشكلة**: مستخدم عادي يحاول الوصول إلى مسارات خاصة بالمدير لتعديل الرواتب.
- **الحل المُطبق**: تم تطبيق (Role-Based Access Control - RBAC). على مستوى قاعدة البيانات وعلى مستوى الـ (API)، يتم فحص الـ `role` المرفق مع توكن الـ JWT قبل تنفيذ أي استعلام (Query).

### 4. عزل قاعدة البيانات (Network Isolation)
- في بيئة `docker-compose`، قاعدة البيانات `MySQL` تعمل على شبكة داخلية مغلقة، ولا يتم فتح منفذ (Port) لها للخارج، الطريقة الوحيدة للوصول إليها هي من خلال خادم (Flask Backend) فقط.
