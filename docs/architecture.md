# التصميم المعماري - نظام إدارة الموظفين (بعد التحويل الجذري)

## 1. نظرة عامة

النظام مبني على معمارية 3 طبقات، بفصل واضح بين كل مسؤولية:

```
backend/
└── app/
    ├── routes/       # طبقة العرض (Controllers) - رفيعة: تستقبل الطلب وتُرجع الرد فقط
    ├── services/      # طبقة منطق الأعمال - كل الحسابات والقواعد هنا
    ├── models/         # طبقة البيانات (SQLAlchemy ORM)
    ├── middleware/    # RBAC + التدقيق (Audit) - تُطبَّق كديكوريتور على أي route
    └── utils/          # التحقق من المُدخلات، رفع الملفات، معالجة الأخطاء
```

هذا يختلف جذريًا عن البنية القديمة حيث كانت كل هذه المسؤوليات (المصادقة
+ البريد + الموظفون + الحضور + الرواتب + المجموعات) متكدّسة داخل كلاس
PHP واحد (`Admin`)، ما جعل من السهل نسيان تحقق صلاحية في نقطة واحدة (كما
حدث فعليًا في `employees/update.php`).

## 2. الكيانات الرئيسية (Database Schema)

| الجدول | الوصف |
|---|---|
| `users` | حسابات الدخول (username, password hash, role) |
| `employees` | بيانات الموظف الشخصية والوظيفية |
| `departments` | الأقسام |
| `attendance` | سجل الحضور اليومي |
| `holidays` / `weekends` | أيام العطل الرسمية وأيام نهاية الأسبوع |
| `login_rules` / `late_fines` | قواعد وقت الدخول وغرامات التأخير |
| `salary_structure` / `payroll` | الراتب الأساسي والرواتب المولَّدة شهريًا |
| `groups` / `group_members` | مجموعات العمل |
| `tasks` / `task_assignments` / `task_progress` / `task_files` | المهام ومتابعتها |
| `documents` | مستندات الموظف (شهادات، خبرات...) |
| **`audit_logs`** | **جديد** — سجل كل عملية حساسة (من، ماذا، متى) |

تعريف كامل الأعمدة موجود في `database/schema.sql`، ويُقابله في الكود
نماذج SQLAlchemy في `backend/app/models/`.

## 3. الأمان (Security)

تم دمج ميزات الأمان التالية في النظام، مطابقةً لما ورد في محاضرات
S-SDLC:

- **المصادقة (Authentication):** JWT (JSON Web Tokens) عبر
  Flask-JWT-Extended، صلاحية الرمز محدودة زمنيًا (30 دقيقة افتراضيًا).
- **التفويض (Authorization):** RBAC عبر ديكوريتور `role_required` يقرأ
  الدور من الـ JWT الموقَّع فقط، أبداً من مُدخلات العميل.
- **تشفير كلمات المرور:** bcrypt (salt تلقائي لكل كلمة مرور).
- **سياسة كلمة مرور قوية:** 12 حرفًا على الأقل + حرف كبير + صغير + رقم
  + رمز (`app/utils/validators.py`).
- **الحد من معدل الطلبات (Rate Limiting):** Flask-Limiter، 5
  محاولات/دقيقة على تسجيل الدخول لمنع هجمات التخمين.
- **رفع ملفات آمن:** قائمة بيضاء لامتدادات الملفات + حد أقصى للحجم (5MB)
  + إعادة تسمية عشوائية (`app/utils/file_upload.py`) بدل قبول أي ملف
  كما كان يحدث سابقاً.
- **سجل التدقيق (Audit Log):** كل عملية كتابة حساسة (إنشاء/تعديل/حذف
  موظف، توليد راتب، تسجيل دخول ناجح/فاشل) تُسجَّل في `audit_logs` عبر
  `app/middleware/audit.py`.
- **الأسرار في متغيرات بيئة فقط:** لا يوجد أي كلمة مرور أو مفتاح مكتوب
  داخل الكود؛ كل شيء عبر `.env` (غير مرفوع لـ Git) — إصلاح مباشر لمشكلة
  بيانات قاعدة البيانات وكلمة مرور Gmail المكتوبة نصًا صريحًا في النسخة
  القديمة.

## 4. النشر (Deployment)

يُستخدم Docker لتغليف الواجهة الخلفية والأمامية وقاعدة البيانات، ويتيح
`docker-compose.yml` تشغيل جميع الخدمات بأمر واحد: `docker compose up`.

## 5. التكامل والنشر المستمر (CI/CD)

يستخدم المشروع GitHub Actions (`.github/workflows/ci.yml`) لأتمتة: فحص
جودة الكود (flake8)، وتشغيل جميع الاختبارات (وحدة + تكامل + أمان)، وبناء
الواجهة الأمامية — تلقائيًا مع كل push أو pull request، قبل أي دمج على
الفرع الرئيسي.

## 6. الخلاصة

توفّر هذه البنية أساسًا قويًا وقابلًا للتوسّع، مع تركيز واضح على الفصل
بين الاهتمامات وقابلية الاختبار والأمان منذ البداية (Security by
Design)، بدل إضافته لاحقًا.
